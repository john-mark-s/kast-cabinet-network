## Project Overview

Build **Kast Cabinet Network** — an open-source transparency dashboard that maps the financial, corporate, and lobbying connections of Chile's 24 cabinet ministers (inaugurated March 11, 2026). The tool cross-references public data from ChileCompra (procurement), InfoLobby (lobbying meetings), InfoProbidad (asset/interest declarations), and SII (corporate registry) using the **RUT** (Chile's universal tax ID) as the linking key.

### Architecture

Fully serverless. Python scrapers run as GitHub Actions on cron schedules, producing static JSON files. A React (Vite + TypeScript) frontend on GitHub Pages consumes those JSON files. No backend server, no database — just flat files in the repo. Git history is the audit trail and diff engine.

### Three core views

1. **🔗 Red de conexiones** — Interactive force-directed graph showing entities and connections with red flag indicators
2. **📡 Cambios recientes** — Weekly changelog showing what's new or changed, auto-generated by diffing graph.json against previous git commit
3. **📎 Fuentes y verificación** — Source directory + step-by-step verification guide + legal basis, so any visitor can independently verify every data point

### Phase transitions

- **Phase 1 (manual seed):** Static graph.json built from config.py + manual research. All 24 ministers listed, Quiroz fully mapped as proof-of-concept.
- **Phase 2 (auto):** Scrapers detect when ministers appear in InfoLobby (~late March), InfoProbidad (~April 10), and ChileCompra. New data auto-commits to repo, triggers graph rebuild + changelog generation. **This transition happens automatically** — the scrapers run on schedule and commit data when they find new results.
- **Phase 3+:** Graph expands as scrapers discover new company RUTs in declarations, which feed back into ChileCompra queries, which surface more connections.

### 2a. ChileCompra (Procurement)

**OCDS Bulk Data** (historical, no auth required):
- Downloads: https://datos-abiertos.chilecompra.cl
- .jsonl.gz, CC0, tenders from 2009+
- **Preferred approach for Phase 1**: download bulk files and filter by RUT locally

**REST API** (incremental):
- Base: `https://api.mercadopublico.cl/servicios/v1/publico/`
- Public test key: `F8537A18-6766-4DEF-9E59-426B4FEE2844` (rate-limited, use for development)
- Production key: requires registration at api.mercadopublico.cl — **may require Chilean RUT/citizenship**. Use the test key first; if it proves insufficient, fall back entirely to bulk OCDS data. Do NOT block the project on obtaining a production key. Store any key as `CHILECOMPRA_API_KEY` in GitHub Secrets and read via `os.environ.get("CHILECOMPRA_API_KEY", "F8537A18-6766-4DEF-9E59-426B4FEE2844")`.
- Supplier lookup: `Empresas/BuscarProveedor?rutempresaproveedor={RUT_WITH_DOTS_AND_DASH}&ticket={key}` — RUT must have dots+dash: `96.953.120-8`
- Supplier orders: `ordenesdecompra.json?CodigoProveedor={code}&ticket={key}`
- Full API docs: https://github.com/MartinGonzalezAlvarez/api_mercadop

```python
# scripts/scrape_chilecompra.py
"""
1. Load all company RUTs from config.py (known_companies across all ministers)
   ALSO load any RUTs written to data/processed/infoprobidad/discovered_ruts.json
   by the InfoProbidad scraper (this is the AUTO-EXPANSION mechanism).
2. For each RUT: call BuscarProveedor → get CódigoEmpresa
3. Query ordenesdecompra for that code
4. For each order: buyer org, amount, date, type (licitación/trato directo), description
5. Also query by buyer org code for each ministry to find suppliers not yet in config
6. Flag "trato directo" as red flag
7. Save to data/processed/chilecompra/{rut_normalized}.json
"""
```

**AUTO-EXPANSION mechanism**: `scrape_infoprobidad.py` writes newly discovered company RUTs to `data/processed/infoprobidad/discovered_ruts.json`. `scrape_chilecompra.py` reads that file at startup and adds those RUTs to its query list. This is how the graph grows organically — no writes back to `config.py` needed.

## Step 3: Graph Builder + Changelog Generator

### 3a. build_graph.py

Reads config.py + all processed data → unified graph.json.

**Node types:** person, company, institution, corporate_client
**Edge types:** owned, heads, family, works_at, contract, advisory, met_with, received_gift, board, govt_role, rumored

**Red flags:**
1. `SELF_DEALING` — minister heads institution that contracts with their company
2. `REVOLVING_DOOR` — minister/spouse worked at agency that contracted their firm
3. `DIRECT_AWARD_CLUSTER` — company gets high % of trato directo
4. `LOBBY_TO_CONTRACT` — org lobbies minister, then wins contract within 90 days
5. `CONCENTRATION` — single supplier >20% of ministry procurement
6. `CORPORATE_RESTRUCTURING` — company changes coinciding with campaign/appointment
7. `GOV_CONTRACTOR` — minister-linked company has active gov contracts
8. `COLLUSION_LINK` — company advised entities convicted of collusion
9. `ACTIVE_CONTRACT` — contract ongoing when minister takes office

**Critical:** Every node and edge must carry:
- `sources[]` — list of source keys from SOURCES dict
- `verifyUrls[]` — direct links for user to verify that specific data point
- `addedDate` — when this entity first appeared in the graph
- `lastModified` — when data about this entity last changed

**Output:** `data/graph/graph.json`

### 3b. diff_graph.py — Changelog Generator

Compares current graph.json vs previous (from `git show HEAD~1:data/graph/graph.json`).

**Detects:** NEW_NODE, REMOVED_NODE, NEW_EDGE, CHANGED_EDGE, NEW_FLAG, RESOLVED_FLAG, CORPORATE_EVENT, DECLARATION_FILED, MINISTER_APPEARED (in InfoLobby/InfoProbidad for first time)

Each changelog entry carries source attribution and node references that link to the graph view.

**Output:** `data/graph/changelog.json` grouped by ISO week.

**Phase 1 (Week 1): Static seed + full frontend**
- config.py with all 24 ministers (Quiroz fully mapped, others as seed)
- Manual graph.json with Quiroz network (~21 nodes, ~22 edges)
- Manual changelog.json with seed events
- Frontend with all three views + minister selector
- Deploy to GitHub Pages
- SHAREABLE AND NEWSWORTHY immediately

**Phase 2 (auto-triggered, ~late March to April):**
- Scrapers run weekly on schedule
- When InfoLobby returns results for a minister → auto-commit → changelog: "Minister now registered"
- When InfoProbidad declaration appears → extract company RUTs → feed to ChileCompra scraper
- Each new company RUT expands the graph organically
- No manual intervention needed — git commits trigger rebuild + deploy

**Phase 3 (Week 4+):**
- All ministers should have InfoProbidad declarations by ~April 10
- Full cross-referencing: declarations ↔ procurement ↔ lobby meetings ↔ corporate registry
- Red flags start firing across the full cabinet
- Changelog becomes the primary news-generating feature

**Phase 4 (Ongoing):**
- Expand to subsecretaries, regional delegates
- Add BCN legislative data
- Campaign finance (requires SERVEL PDF scraping)
- Community contributions for minister research

## Key Gotchas

1. **RUT formatting**: Normalize to dashed-no-dots (12345678-9) internally. ChileCompra API needs dots+dash (12.345.678-9). Validate with módulo 11 check digit.

2. **Phase 2 is automatic**: Scrapers run on cron. When they find new data, they commit it. The build-graph workflow triggers on data/ changes. The deploy workflow triggers on frontend/ changes. The entire pipeline is hands-off after Phase 1.

3. **Every data point needs sources + verify URLs**: This is non-negotiable. The tool's credibility depends on users being able to verify everything independently. If a data point can't be sourced, it doesn't go in the graph.

4. **InfoLobby pagination**: 100/page. Always follow.

5. **GitHub Pages limits**: 1GB site, 100GB/month bandwidth. Keep data/graph/ under 50MB.

6. **Git = audit trail**: Each scraper commit captures data state. `diff_graph.py` uses `git show HEAD~1:data/graph/graph.json` to diff.

7. **Defensive coding**: Gov APIs go down. Cache last good response. Log failures to meta.json. Don't fail whole workflow on one timeout.

8. **Privacy**: Only use RUTs from official transparency portals. Never publish private RUTs.

9. **Framing**: "Los patrones en registros públicos son señales para investigación adicional, no evidencia de irregularidades."

10. **Timing**: Ministers take office March 11. InfoLobby ~late March. InfoProbidad ~April 10. ChileCompra historical available immediately. The scrapers handle timing gracefully — they just return empty until data appears, then auto-expand.
